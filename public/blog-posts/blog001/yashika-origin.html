Why Use ChatGPT When I Can Build My Own?

The Origin Story of Project Y.A.S.H.I.K.A.

AUTHOR: HEMANT_DUA DATE: 2026.03.08 CLASS: PUBLIC

Most AI assistants today rely entirely on the cloud. When you ask a question, your request travels to a remote server where a model processes it and sends the response back. The experience feels seamless, but it also means something important: the intelligence you interact with isn’t really yours.

Your data leaves your machine, the model runs somewhere you cannot control, and the system stops working the moment the internet disappears.

That idea always felt limiting.

I started wondering what it would look like if the intelligence itself lived on my machine. Not just a chatbot interface, but something closer to a personal AI system that could understand context, remember interactions, and interact directly with the computer it runs on.

That idea became Project Y.A.S.H.I.K.A. — Yet Another Super Human Intelligent Knowledge Agent.

The goal of the project is simple: build a locally hosted AI assistant that can operate independently of external AI services while still being capable enough to interact with real workflows.

The Spark Behind the Idea

While exploring local AI tools, I came across a video explaining how large language models can run entirely on personal machines instead of relying on cloud APIs.

You can watch it here:

The video demonstrates how tools like Ollama allow developers to run language models locally and interact with them directly. Seeing a model operate completely offline made one thing clear: if the model can run locally, then the rest of the system around it can be designed locally as well.

That realization pushed the idea further. Instead of just running a model in a terminal, I wanted to build something with structure — a system that could manage memory, handle commands, and eventually evolve into a broader personal AI ecosystem.

Why Local AI Matters

Cloud AI services are powerful, but they come with trade-offs. They depend on internet connectivity, rely on external infrastructure, and operate in environments that users cannot fully control.

Running AI locally changes the relationship between the user and the system.

When the model lives on your machine, the data never leaves your device. The assistant can continue functioning even when offline, and it becomes possible to integrate AI directly with the operating system, applications, and hardware devices.

Local AI shifts the system from something you access to something you own.

YASHIKA was designed around that principle.

What YASHIKA Actually Does

YASHIKA is a locally hosted AI system that combines an LLM, persistent memory, system commands, and multi-interface clients into a personal AI assistant.

At its core, YASHIKA acts as an orchestration layer between a user interface, a locally hosted language model, and the computer itself.

The assistant communicates with language models hosted through Ollama, which exposes an API for running models like Gemma or LLaMA locally.

Instead of forwarding prompts to external AI APIs, the system sends requests directly to the model running on the machine.

However, a useful assistant requires more than just model responses. To maintain continuity in conversations, YASHIKA maintains its own structured memory system. Conversations are stored in JSON files that separate long-term memory, active context, and detailed logs. This layered approach allows the system to keep track of recent interactions while preserving historical information for later use.

The assistant can also retrieve information from the web when necessary. If a request requires external information, the system can trigger a DuckDuckGo search, fetch the results, and summarize them locally using the model before returning the response.

Another capability built into YASHIKA is system interaction. Certain commands allow the assistant to open applications, access directories, or launch tools on the system. To ensure safety, only predefined commands are allowed, preventing the assistant from executing arbitrary instructions.

This combination of capabilities transforms the assistant from a simple chat interface into something closer to a system companion.

Architecture of the System

The core of YASHIKA is a backend engine built using FastAPI. This engine acts as the central coordinator that connects the user interfaces, the memory system, and the local language model.

Fig_1: Architecture of YASHIKA

Whenever a user sends a message, the request first reaches the FastAPI server. The server retrieves relevant context from memory, injects it into the prompt, and forwards the request to the model running through Ollama. Once the model produces a response, the backend processes the output, updates the conversation logs, and streams the result back to the user interface.

Separating the orchestration layer from the model layer makes the system easier to extend. New interfaces, features, or models can be added without changing the entire architecture.

Interfaces for Interaction

Currently, YASHIKA can be accessed through two different interfaces.

The first is a web interface built with HTML, CSS, and JavaScript. It provides a simple browser-based chat environment for interacting with the assistant.

The second interface is a desktop application built using GTK and Python. This client is designed to integrate more closely with the system, providing a dashboard-like interface that can eventually support automation triggers, hardware monitoring, and deeper system control.

Both interfaces communicate with the same FastAPI backend, ensuring that the assistant behaves consistently regardless of how the user interacts with it.

Where the Project Is Going

YASHIKA is still an evolving system.

The current version focuses primarily on establishing the foundation for local AI interaction: a working architecture that combines language models, memory systems, command execution, and multiple user interfaces.

Future development will explore several new capabilities. Offline speech recognition and text-to-speech will allow more natural interactions. Bluetooth connectivity could enable integration with wearable devices and sensors. Event-based triggers could allow the assistant to respond automatically to system changes or environmental signals.

The long-term goal is to move beyond a desktop assistant and build something closer to a personal AI ecosystem.

Final Thoughts

Running a language model locally is relatively straightforward today. The real challenge is designing everything around it — context management, memory handling, system integration, and user interfaces.

YASHIKA is an attempt to explore that problem space.

What began as curiosity about local AI infrastructure has grown into a modular system that can interact with models, remember conversations, execute commands, and integrate with the environment it runs in.

The project is still evolving, but each step moves closer to the same idea that started it all:

A personal AI that runs on your machine, understands your environment, and belongs entirely to you.
